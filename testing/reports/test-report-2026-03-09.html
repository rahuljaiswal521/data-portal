Fixes Applied
Race-condition fix: single-TC persist now creates a new result file
Root Cause
When a user clicked the per-row Run button while a full suite run was in progress (or had just been cancelled), _persist_single_tc_result would write back to the same {run_id}.json file that the background _execute_suite_async thread was still writing to. The background thread would overwrite the single-TC result, causing SWR to fetch a stale CANCELLED status after revalidation.
Fix Description
The method now reads the latest result (if any), merges the new TC result into it, assigns a fresh uuid.uuid4() as the run_id, and writes to a new file. Since get_latest_result() returns the file with the newest mtime, SWR always picks up the persisted single-TC result. The background thread continues writing to the old file, which is now harmlessly stale.
Affected File
portal/backend/app/services/testing_service.py
Test assertions updated to match new new-file-per-persist behaviour
Root Cause
Two existing tests assumed in-place mutation (one file, same run_id). After the service change they would fail because a second file is now created on every persist call.
Fix Description
test_merges_result_into_existing_run_when_tc_already_present: assertion changed from len(files)==1 to len(files)==2; added assert data["run_id"] != existing.run_id; reads newest file by mtime.

test_appends_result_when_tc_not_in_existing_run: assertion changed to len(files)==2; reads max(files, key=lambda p: p.stat().st_mtime) to get the newly written file.
