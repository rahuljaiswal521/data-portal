TC004 Late-Arriving Data — Does Not Overwrite Newer Version Positive ✓ Passed

What This Test Validates

Verifies that the late-arriving-data guard prevents out-of-order records from overwriting a more recent version. When a record arrives with an updated_at timestamp older than the currently stored version, the pipeline must skip the update entirely — the current row remains unchanged.

Why It Matters

In real ingestion pipelines, events frequently arrive out of order due to network delays, CDC replay, or reprocessing. Without this guard, a 2020-dated record re-ingested in 2026 would silently overwrite the 2024 current version — corrupting the timeline. The guard (source.updated_at >= target.updated_at) in the MERGE SQL prevents this.

TC005 Null Primary Key — Quarantined to Dead Letter Negative ✓ Passed

What This Test Validates

Verifies that records with a null primary key are blocked from entering the bronze table and instead quarantined to dev.bronze_meta.dead_letter_crm_customers. The null-PK guard prevents unidentifiable records from corrupting downstream joins and SCD2 history.

Why It Matters

A null primary key is fundamentally unresolvable for SCD2 — there is no key to match on for future updates or deletes. Allowing it into bronze silently creates ghost rows. The quality adapter's null-PK filter catches this at ingestion time, before the writer is ever called.

Test Data Scenario

Table truncated before the run. tc005_null_pk.json contains one record with "customer_id": null and otherwise valid fields. The quality adapter identifies it via col("customer_id").isNull(), routes it to quarantine(), and the writer receives an empty good-data frame.

Root Cause & Fix

The quarantine_threshold_pct defaults to 10%. With 1 bad record out of 1 total (100%), the engine's circuit-breaker was raising RuntimeError before calling quarantine(). Fix: _build_test_yaml() in testing_service.py now overrides the threshold to 100 in the test YAML, so negative tests never trip the production safeguard.
