🏗️ What is the Data Platform Portal?

The Data Platform Portal is a self-service web application that lets the team configure, deploy, and monitor data ingestion pipelines — without writing a single line of code. It acts as a friendly front-end sitting on top of the powerful Bronze + Silver data framework already running in Databricks.

Think of it as a control panel: instead of editing configuration files and running scripts directly, team members use a clean web interface to point-and-click their way to production-ready data pipelines.

🔌

Connect Data Sources

Hook up files, databases, APIs and event streams through a guided UI — no YAML expertise needed.

🚀

Deploy to Databricks

One click pushes a fully configured ingestion job to the Databricks workspace in the cloud.

🤖

AI Pipeline Builder

Describe what data you need in plain English — the AI assistant generates the configuration for you.

🧪

Built-in Testing

Run automated test suites against your pipeline before going live — catch problems early.

📊

Monitor & Audit

See run history, data quality scores, and audit logs — all from one dashboard.

🔐

Secure by Default

Microsoft login protects every page. Secrets are stored safely — never in the code.

🚢 How Code Gets to Azure

When a developer makes changes to the portal code, here is the path those changes travel — from a laptop to the live application used by the team.

1
💻 Developer makes changes on their laptop
Code is edited locally — new features, bug fixes, config tweaks. The developer tests them using Docker Compose (a local copy of the whole system) before touching anything live.
2
🔨 Run the redeploy script
A single command kicks off the whole build-and-publish process. The script packages the application into a Docker image — like zipping everything into a self-contained capsule.
bash portal/redeploy.sh [backend | frontend | both]
3
📤 Image pushed to Azure Container Registry
The freshly built Docker image is uploaded to dataportalacr.azurecr.io — Azure's private storage for application images. This is like uploading a new version to an app store.
4
🔄 Azure pulls the new image and restarts
Azure Container Apps notices the new image in the registry, pulls it down, and gracefully swaps the running container with the new version. Existing data is untouched — it lives on the File Shares.
⏱ ~1–2 minutes downtime
5
✅ Team sees the update
Refresh the browser and the new version is live. No server maintenance, no manual restarts — the cloud handles all of that automatically.

🔐 Security & Access

Currently Offline
🚪

Portal Access Status

The portal URL is currently disabled (ingress turned off). Visiting the URL will return a connection error — this is intentional. To bring it online, run bash portal/auth-enable.sh followed by the ingress enable command.

🛡️

Microsoft Login (EasyAuth)

When the portal is online, a Microsoft identity layer sits in front of the entire application. Users are redirected to a Microsoft login page before they can see anything. Only approved accounts get through — like a bouncer at the door.

🔒

Secrets — Never in Code

Sensitive credentials (Databricks token, Anthropic API key) are stored as Azure Container Apps Secrets — a secure vault separate from the application code. Even if someone read the source code, they would not find any passwords.

🔐

Encrypted in Transit

All communication between the browser and Azure uses HTTPS (TLS encryption). All calls from the backend to Databricks are also encrypted. Data is never sent in plain text across the internet.

🛠️ Management Scripts

All day-to-day operations are handled through shell scripts in the portal/ directory. Run these from the root of the project on your laptop.

Script Command What it does
redeploy.sh bash portal/redeploy.sh [backend|frontend|both] Builds a new Docker image and pushes it to Azure — the standard way to ship code changes to production.
auth-enable.sh bash portal/auth-enable.sh Turns on Microsoft login (EasyAuth). Run this before making the portal URL public to the team.
auth-disable.sh bash portal/auth-disable.sh Turns off Microsoft login — useful during development or debugging when you need unauthenticated access.
azure-deploy.sh bash portal/azure-deploy.sh Full from-scratch deployment — provisions all Azure resources and deploys both containers. Use this to set up a brand-new environment.
docker-compose.yml docker compose -f portal/docker-compose.yml up Starts the entire portal locally on your laptop for development and testing — no Azure required.

⚖️ Local Development vs Production

The portal can run in two modes. Local is for developers building and testing new features. Production is the live environment the team actually uses.

💻

Local Development

Runs on a developer's laptop

How to start
docker compose up
Runs on
Your laptop — uses local ports (e.g. localhost:3000)
Config files
Local folders on your laptop's hard drive
Auth
No login required — open access for easy testing
Purpose
Develop and test new features safely before deploying
Availability
Only while Docker is running on your machine
☁️

Production (Azure)

Live environment for the whole team

How to start
bash portal/auth-enable.sh + ingress enable
Runs on
Azure Container Apps — public HTTPS URL
Config files
Azure File Shares — persisted in the cloud
Auth
Microsoft login required — only approved accounts
Purpose
The real thing — used by the full team for live pipelines
Availability
24/7 — Azure manages uptime and scaling automatically
